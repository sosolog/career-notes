# 단일 서버 Docker 환경에서의 Blue/Green 무중단 배포

## 배경 — 왜 이 구조가 필요했나

첫 번째 프로젝트에서 수동 배포를 자동화하기 위해 GitHub Actions + CodeDeploy 기반 파이프라인을 먼저 구성했습니다.
이 시점에는 무중단까지는 구현하지 않았습니다.

이후 다른 프로젝트에서 문제가 생겼습니다.
해당 프로젝트는 공장 현장이 아닌 **전사 직원이 상시 사용하는 내부 시스템**이었습니다.
배포할 때마다 5~30초 가량 서비스가 단절되었고,
배포를 하려면 전사 공지를 하거나 특정 배포 시간을 따로 확보해야 했습니다.
결국 배포를 자유롭게 할 수 없는 것이 운영 부담으로 이어졌고,
서비스 중단 없이 배포할 수 있는 구조가 필요했습니다.

---

## 제약 조건 — 왜 Kubernetes가 아니었나

엔터프라이즈 환경이었다면 Kubernetes가 자연스러운 선택이었겠지만,
당시 환경에서는 현실적으로 불가능했습니다.

- **단일 서버, 내부망 환경** — 멀티 노드 클러스터 구성 자체가 불가
- **인프라 운영 인력 없음** — 운영은 개발자 1인 중심, Kubernetes 운영 부담을 감당할 수 없음
- **비용 제약** — 추가 서버 비용 없이 해결해야 함
- **외부 Load Balancer 없음** — Nginx + Docker만 사용 가능한 환경

주어진 환경에서 현실적인 최적해가 필요했습니다.

---

## 왜 Blue/Green이었나

**Rolling Update는 단일 서버에서 적합하지 않았습니다.**

- replica 1이면 Rolling 자체가 불가
- replica 2 이상으로 구성하면 단일 서버에서 리소스 부담 증가
- 컨테이너 교체 방식은 어떻게 해도 순간 단절 발생

**Blue/Green을 선택한 이유:**

- 새 버전이 완전히 기동된 것을 확인한 뒤 트래픽 전환 가능
- 전환 실패 시 Nginx upstream을 되돌리는 것만으로 즉시 롤백 가능
- DB 마이그레이션 타이밍을 배포와 분리하여 통제 가능

단일 서버 환경에서 추가 비용 없이 구현할 수 있는 가장 안전한 무중단 유사 구조였습니다.

---

## 구현 방식

Nginx upstream을 포트 스위칭으로 전환하는 방식을 적용했습니다.

```
Blue:  포트 8081 (현재 운영 중)
Green: 포트 8082 (신규 버전 기동)

배포 흐름:
Green 컨테이너 기동
       ↓
Health check 통과 확인
       ↓
Nginx upstream → Green(8082)으로 전환 (nginx reload)
       ↓
Blue 컨테이너 일정 시간 유지 후 종료
```

Nginx reload는 무중단이지만, upstream 변경 직후 기존 커넥션 처리를 고려하여
Blue 컨테이너를 즉시 종료하지 않고 일정 시간 유지했습니다.

## 운영 중 발생한 문제 — DRM 라이브러리 크래시

Health check를 통과했는데도 배포 후 서비스가 죽는 현상이 발생했습니다.

**원인**

애플리케이션에 외부 DRM 라이브러리가 연결되어 있었습니다.
애플리케이션 자체는 정상적으로 기동되지만,
최초 DRM 라이브러리 호출 시 JNI 라이브러리의 Docker volume 바인딩이
제대로 되지 않은 경우 크래시가 발생하면서 컨테이너 프로세스가 완전히 죽었습니다.

**문제점**

Health check는 애플리케이션 기동 여부만 확인했기 때문에,
DRM 크래시로 프로세스가 죽어도 이를 감지하지 못했습니다.
무중단 배포 구조 안에서 죽은 프로세스를 자동으로 복구하는 방법이 없었습니다.

**해결 — Python 헬스체크 프로세스 추가**

별도의 Python 헬스체크 프로세스를 작성했습니다.

```
5초마다 애플리케이션 응답 확인
       ↓
응답 없으면 컨테이너 재기동 시도
       ↓
3회 이상 정상 복구 실패 시 → Slack 알림 발송
```

단순 HTTP 응답이 아니라 실제 서비스 동작 여부를 주기적으로 확인하고,
자동 복구를 시도하되 복구 실패 시 즉시 알림이 오도록 했습니다.

---

## 운영하면서 배운 것

**Health check는 단순 200 응답으로 충분하지 않습니다.**
DB 연결, 주요 의존 라이브러리 초기화 여부까지 확인해야 실제 장애를 잡을 수 있습니다.
DRM 크래시 사례가 이를 직접 경험하게 해줬습니다.

**세션은 배포 구조와 분리해야 합니다.**
Tomcat 메모리 기반 세션은 컨테이너 재기동 시 손실되어 배포 중 세션 끊김이 발생했습니다.
근본적인 해결을 위해 JWT 방식으로 전환하여 세션 문제를 해소했습니다.

**DB 스키마 변경과 배포는 분리해야 합니다.**
배포 중 스키마가 바뀌면 Blue/Green 전환 타이밍에 문제가 생길 수 있습니다.
Expand → Deploy → Contract 전략의 필요성을 체감했습니다.

**무중단 배포는 기술이 아니라 설계 사고입니다.**
인프라 규모와 상관없이 트래픽 스위칭 구조와 롤백 전략은 동일한 패턴으로 적용됩니다.
Kubernetes 없이도 같은 개념을 구현하면서 이 패턴을 체득했습니다.

---

## 한계 및 트레이드오프

- Python 헬스체크 프로세스가 별도로 존재하면서 관리 포인트가 하나 추가됐습니다.
- 단일 서버이기 때문에 서버 자체가 내려가면 이 구조 전체가 의미 없어집니다. 고가용성은 확보되지 않습니다.
- Kubernetes 환경이었다면 헬스체크, 자동 복구, 롤링/블루그린 전략을 플랫폼 수준에서 해결할 수 있었겠지만,
  당시 환경의 제약 속에서 직접 구현하면서 오히려 내부 동작 원리를 깊이 이해하게 됐습니다.
