# IoT 텔레메트리 처리 지연 개선

## 배경

MQTT 브로커로부터 두 종류의 데이터를 수신하고 있었습니다.

- **Attributes** — 기기 메타데이터 (기기명, 타입 등)
- **Telemetry** — 실시간 측정값 (counter, 전력 등)

두 데이터는 하나의 토픽으로 들어오며, 수신 시 분리하여 각각의 테이블에 저장합니다.
단순 저장에서 끝나지 않고, 저장된 값을 기반으로 **설비 상태 계산 및 Stop 구간 생성/업데이트**가 뒤따랐습니다.

---

## 문제

초기 구조는 메시지를 **건별로 즉시 처리**하는 방식이었습니다.

```
MQTT 수신 → Attribute/Telemetry 분리 → Insert/Update
         → 상태 계산 (Java단 SELECT/UPDATE 반복)
         → Stop 구간 생성/업데이트
```

이 구조에서 DB와의 왕복(roundtrip)이 메시지 하나당 반복적으로 발생했고,
처리 속도보다 수신 속도가 빠른 상황이 지속되면서
**텔레메트리 저장 지연이 점진적으로 누적되어 최대 30분까지 벌어졌습니다.**

---

## 시도 1 — 병렬 처리 (Executor 다수 도입)

처리량을 높이기 위해 여러 개의 Executor로 병렬 처리를 시도했습니다.

**결과: DB Deadlock 빈도 증가로 포기**

상태 계산과 Stop 구간 업데이트 로직이 동일한 행을 동시에 접근하면서
Deadlock이 빈번하게 발생했습니다.
병렬화는 처리량을 높이기 전에 정합성 문제를 먼저 만들었습니다.

---

## 시도 2 — 단일 Executor + DB 큐 배치 처리

Deadlock을 피하기 위해 **단일 Executor**로 순서를 보장하면서,
단순 Insert 구간은 **MyBatis 배치 모드**로 묶어서 처리했습니다.

```
MQTT 수신 → DB 큐 적재
         → 단일 Executor가 배치로 꺼내서 처리
         → 단순 Insert: MyBatis 배치 모드 (roundtrip 감소)
         → 상태 계산/Stop 구간: 여전히 Java단 SELECT/UPDATE 반복
```

**결과: 부분 개선, 근본 해결 안 됨**

단순 Insert 구간은 빨라졌지만,
상태 계산과 Stop 구간 업데이트는 SELECT/UPDATE가 혼재되어 있어
MyBatis 배치 모드 적용이 불가능했습니다.
이 부분의 roundtrip이 여전히 병목이었습니다.

---

## 시도 3 — 상태 계산 로직을 Procedure로 이관

병목의 원인이 **Java ↔ DB 왕복 횟수** 자체라는 것을 확인한 뒤,
SELECT/UPDATE가 혼재된 로직을 DB 내부에서 처리하도록 **Procedure로 이관**했습니다.

```
MQTT 수신 → DB 큐 적재
         → 단일 Executor 배치 처리
         → 단순 Insert: MyBatis 배치 모드
         → 상태 계산/Stop 구간: Procedure 호출 (DB 내부 처리)
```

Java단에서 여러 번 왕복하던 부분이 Procedure 호출 한 번으로 줄었습니다.

**결과: 저장 지연 30분 → 1.5분으로 개선**

---

## 추가 대응 — Stop 구간 누락/중복 검증

Procedure 이관 과정에서 **Stop 구간 누락 및 중복**이 발생했습니다.
고정 Stop / 비고정 Stop이 혼재된 계산 로직이 이관 중 일부 케이스를 놓친 것이었습니다.

실시간 데이터로는 재현이 어려워 **과거 데이터를 리플레이하는 테스트 환경**을 구성하고,
케이스별로 로직을 검증하며 수정했습니다.

---

## 정리

| 시도 | 방법 | 결과 |
|------|------|------|
| 1 | Executor 다수 (병렬 처리) | Deadlock 증가 → 포기 |
| 2 | 단일 Executor + MyBatis 배치 모드 | 부분 개선, 구간 업데이트 여전히 병목 |
| 3 | 상태 계산 로직 Procedure 이관 | 지연 30분 → 1.5분 |

병렬화가 항상 답이 아니었고,
병목의 위치를 정확히 파악한 뒤 **roundtrip 자체를 줄이는 방향**으로 접근한 것이 핵심이었습니다.

---

## 한계 및 트레이드오프

- Procedure에 비즈니스 로직이 들어가면서 **Java 코드와 DB 로직이 분리**되었습니다. 이후 로직 변경 시 Procedure를 직접 수정해야 하는 유지보수 부담이 생겼습니다.
- 큐 도입 없이 단일 Executor로 처리하기 때문에, 수신량이 크게 증가하면 다시 지연이 발생할 수 있는 구조입니다.
- 당시 환경(소규모 팀, 빠른 운영 안정화 필요)을 고려한 선택이었으며, 규모가 커진다면 메시지 큐 도입과 함께 구조를 재검토할 필요가 있습니다.
