# 메일 / 메시지 중계 서비스 아키텍처 설계

## 배경

여러 프로젝트에서 메일·메시지 발송 로직이 각자 구현되어 있었습니다.
발송 계정 관리, 인증, 재시도 로직이 프로젝트마다 중복되었고,
운영 중 발송 실패가 생겨도 추적이 어려운 상태였습니다.

이를 중앙화된 중계 서비스로 통합하는 것이 목표였습니다.

---

## 초기 설계 — Client System 단위로 전부 묶기

처음에는 단순하게 **Client System 하위에 Sender / Scheduler / Agent를 전부 두는 구조**를 생각했습니다.

```
Client System A
├── Sender (발송 계정)
├── Scheduler (발송 주기)
└── Agent (연결)

Client System B
├── Sender
├── Scheduler
└── Agent
```

구현하기 쉽고 각 시스템이 독립적으로 관리된다는 장점이 있었습니다.

---

## 문제 인식 — 중복과 부하

설계를 진행하면서 두 가지 문제가 보였습니다.

**1. 동일 주기 스케줄러 중복**

여러 시스템이 동일한 주기로 발송 작업을 돌릴 수 있는데,
시스템마다 Scheduler를 독립적으로 두면
같은 주기의 스케줄러가 시스템 개수만큼 생겨 불필요한 부하가 발생합니다.

**2. SMTP 서버 설정 중복**

SMTP 서버 자체는 여러 시스템이 공유할 수 있는 인프라인데,
시스템마다 각자 설정을 관리하면 동일한 설정이 중복으로 존재하게 됩니다.

---

## 개선된 설계 — Global / Client System 분리

중복되는 부분은 **Global**로 올리고, 시스템별로 달라야 하는 부분만 **Client System** 하위에 두었습니다.

```
Global
├── Scheduler (주기 공유 — 여러 시스템이 동일 스케줄러 재사용)
└── SMTP 서버 설정 (서버 인프라 공유)

Client System A
├── Sender A-1 (a@company.com — A 사업부 발신 계정)
├── Sender A-2 (b@company.com — B 사업부 발신 계정)
└── Agent (Sender ↔ Scheduler 조합 관리)

Client System B
├── Sender B-1
└── Agent
```

**Sender는 왜 Client System 하위에 두었나**

같은 시스템 안에서도 사업부마다 다른 발신 계정을 써야 할 수 있습니다.
SMTP 서버 설정(인프라)은 공유하되, 실제 발신 계정 관리는 시스템 하위에서 하도록 분리했습니다.

---

## Agent — Sender와 Scheduler를 연결하는 다리

Sender와 Scheduler의 조합은 여러 가지가 될 수 있습니다.

- Sender A-1 + 매일 오전 9시 스케줄러
- Sender A-2 + 매주 월요일 스케줄러

이 조합을 **Agent가 label로 관리**합니다.
API 호출 시에는 label만 넘기면, 해당 label에 연결된 Sender와 Scheduler로 발송이 이루어집니다.

```
API 요청 → label: "weekly-report"
         → Agent 조회 → Sender A-1 + 매주 월요일 스케줄러
         → 발송 처리
```

호출하는 쪽에서는 Sender나 Scheduler의 구체적인 설정을 알 필요 없이
label 하나로 발송 요청을 할 수 있습니다.

---

## 최종 구조 요약

```
[API 요청 (label)]
       ↓
    Agent
   ↙     ↘
Sender   Scheduler
  ↓          ↓
발신 계정   발송 주기
  ↓
SMTP 서버 (Global)
```

| 컴포넌트 | 범위 | 역할 |
|----------|------|------|
| Scheduler | Global | 발송 주기 관리, 시스템 간 공유 |
| SMTP 서버 설정 | Global | 서버 인프라 공유 |
| Sender | Client System | 발신 계정 관리 |
| Agent | Client System | Sender ↔ Scheduler 조합, label 관리 |

---

## 발송 상태 전달 — 검토한 방식들

발송 요청은 A 시스템에서 하지만, 실제 발송 상태는 중계 서비스에서만 관리됩니다.
A 시스템이 발송 진행 상태를 알 수 없다는 문제가 있었고, 세 가지 방식을 검토했습니다.

**1. Kafka 기반 이벤트 전달**
상태 변화가 생길 때마다 Kafka로 이벤트를 발행하고, A 시스템이 구독하는 방식입니다.
시스템 간 결합도가 낮고 확장성이 좋지만, 도입사 전산팀이 Kafka를 운영할 수 있는 환경이 아니었습니다.
관리 포인트가 하나 더 늘어나는 것이 부담이었습니다.

**2. A 시스템 DB에 상태 로그 직접 Insert**
중계 서비스가 A 시스템의 DB에 직접 접근하여 상태 로그를 Insert하는 방식입니다.
중계 서비스가 외부 DB에 직접 쓴다는 점에서 의존성이 역방향으로 생기는 문제가 있었습니다.

**3. 중계 서비스 히스토리 테이블을 A 시스템이 직접 Polling**
A 시스템이 중계 서비스의 발송 히스토리 테이블을 직접 조회하는 방식입니다.
도입사 내 프로젝트들이 같은 DB 서버를 공유하고 있어 구현이 단순했고, 결국 이 방식을 선택했습니다.

**결과적으로 3번을 선택했지만, 지금 돌아보면 좋은 선택이 아니었습니다.**
중계 서비스의 내부 DB 구조가 외부에 노출되고, 시스템 간 결합도가 높아지는 문제가 있습니다.
같은 DB 서버를 공유한다는 환경적 이유로 선택한 것이었고,
Kafka나 Webhook 방식이 구조적으로 더 적합했을 것이라고 생각합니다.

---

## 한계 및 트레이드오프

- Global Scheduler를 여러 시스템이 공유하기 때문에, 특정 시스템의 요구사항으로 스케줄러를 변경하면 다른 시스템에 영향을 줄 수 있습니다.
- DB 기반 구조이므로 발송량이 크게 증가하면 DB 부하로 이어질 수 있습니다. 규모가 커진다면 Kafka 도입을 재검토할 필요가 있습니다.
- 현재 구조는 도입사 운영 환경에 맞춰 단순하게 유지한 선택이었습니다.
