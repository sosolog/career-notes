# Career Notes

백엔드 개발을 중심으로,  
서비스 운영에 필요한 인프라 구성부터 배포 자동화까지  
전체 라이프사이클을 경험해온 엔지니어입니다.

작은 조직 환경에서 한정된 리소스 안에서  
실용적인 해결책을 선택하고 이를 직접 구현하는 역할을 해왔으며,  
이 저장소는 그 과정에서의 기술적 판단과 trade-off를 정리하기 위한 공간입니다.

## 🔍 Prometheus 기반 통합 모니터링 시스템 구축

사내 특수 DB 환경에서 필요한 메트릭을 수집하기 위해,  
공식적으로 지원되지 않는 영역을 직접 개발하고 모니터링 아키텍처를 설계했습니다.

**핵심 기술 결정**
- 공식 Exporter가 없는 Tibero DB 환경에서의 모니터링 요구사항 발생
- Oracle Exporter 구조를 분석한 뒤 Java 기반 Custom Exporter 개발
- 시계열 메트릭(CPU, 메모리)과 비시계열 메트릭(실행 중 SQL, 락 정보)의 성격 차이를 고려한 수집 구조 설계
- 프론트엔드의 다수 REST API 반복 호출 문제를 줄이기 위해,
  미들웨어에서 데이터를 통합 수집한 뒤 전달하는 구조로 개선
- Prometheus 서버 정보를 프론트엔드에 직접 노출하지 않도록 중간 계층 구성

**주요 성과**
- 정형화하기 어려운 리스트 형태 데이터(SQL 목록, 락 정보)에 대한 별도 수집 구조 구현
- 프론트엔드의 다중 API 호출을 미들웨어 통합 수집 방식으로 대체하여 부하 감소
- 미들웨어에서 2–5초 주기로 메트릭을 수집·전달하는 구조 구성
- Prometheus 서버 정보가 외부에 노출되지 않는 모니터링 구조 확보

※ 리소스 및 활용 환경의 제약으로 인해 장애 탐지보다는 **운영 상태를 확인하는 용도**로 활용됨

## 📬 메일 / 메시지 중계 서비스 아키텍처 설계

여러 프로젝트에서 반복적으로 구현되던 메일·메시지 발송 로직을 정리하고,  
중앙화된 중계 구조로 통합 관리 체계를 설계·구현했습니다.

**핵심 기술 결정**
- 프로젝트별 발송 계정(Sender), 공통 SMTP 풀, 스케줄러를 Agent 개념으로 연결하여 책임 분리
- 동일 발송 주기를 가진 작업들을 단일 스케줄러로 통합하여 리소스 효율화
- 초기 Kafka 기반 설계를 검토했으나, 시스템 환경을 고려해 DB 기반 구조로 전환
- 외부 메시지 서비스 API 연동 및 JWT, API Key 기반 인증 방식 지원

**주요 성과**
- 프로젝트별로 분산되어 있던 발송 로직 제거 및 중복 개발 방지
- 신규 시스템 추가 시 Sender 설정만으로 연동 가능한 구조 확보
- 발송 실패 추적 및 수동 재시도 기능을 통해 운영 안정성 향상

## 🧩 공통 설정 및 레거시 연동 모듈 개발

여러 프로젝트를 진행하며 반복되는 초기 설정 작업과  
레거시 환경 연동 부담을 줄이기 위해,  
공통 설정과 레거시 대응을 함께 다루는 모듈을 개발했습니다.

**핵심 기술 결정**
- JPA/MyBatis, 다중 DB 설정을 YAML 기반으로 선언하면 필요한 Bean이 자동 등록되도록 구성
- web/jpa/mybatis 의존성을 통합하여 프로젝트 생성 시점부터 기본 환경이 갖춰지도록 설계
- imageView 등 반복적으로 사용되던 공통 컴포넌트 제공
- 설정 파일만으로 동작하도록 구성하여 초급 개발자도 안정적으로 적용 가능하도록 설계

- 레거시 환경에서 사용되던 Nexacro 기반 시스템을 Spring Boot 환경으로 이식
- Handler Argument Resolver 구현으로 Nexacro XML 데이터를 Controller에서 직접 바인딩
- web.xml 기반 Excel 처리 서블릿을 Spring Boot Java Config 방식으로 전환
- 기존 SQL ID + 파라미터 기반 공통 API 구조를 유지하며 최신 기술 스택 적용

**주요 성과**
- 프로젝트 초기 설정 및 레거시 연동에 소요되는 시간 감소
- DB 설정, Bean 등록, 요청 바인딩 과정에서 발생하던 휴먼 에러 사전 방지
- 기존 개발 방식과 최신 Spring Boot 환경 간 호환성 확보
- 여러 프로젝트에 일관된 초기 구조 적용으로 개발 표준화 기여

## 🚀 Docker 기반 무중단 배포 파이프라인 구축

수동 배포 환경을 개선하기 위해,  
여러 서버 환경에서 서비스 중단을 최소화하는 배포 절차를 설계·구현했습니다.

**핵심 기술 결정**
- 서버 환경별 실행 환경 차이를 줄이기 위해
  전체 서비스를 Docker 컨테이너 기반으로 표준화
- GitHub Actions 기반으로 빌드 및 배포 파이프라인을 구성하여
  배포 흐름을 단순화
- 각 서버마다 Docker Compose 기반 구성, Blue/Green 포트 스위칭 방식 적용
- 신규 버전 실행 → 헬스체크 → 트래픽 전환 → 기존 버전 종료 흐름 자동화

**주요 성과**
- 수동 jar/war 배포에서 자동화 배포로 전환하여 휴먼 에러 감소
- 무중단 배포 도입으로 서비스 다운타임 제거
- 비용 효율적인 자체 배포 구조 확보

※ Kubernetes 등 오케스트레이션 도구는 사용하지 않음
